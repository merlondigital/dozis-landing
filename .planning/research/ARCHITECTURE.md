# Architecture Patterns

**Domain:** Event management / QR check-in app (DOZIS.)
**Researched:** 2026-03-29
**Overall confidence:** HIGH

## Recommended Architecture

**Single Cloudflare Worker with static assets** -- the existing React + Vite landing page and the new event app live in one Vite project, deployed as one Worker. The `@cloudflare/vite-plugin` unifies the SPA frontend and the Hono API backend into a single deployment artifact.

```
Browser
  |
  |-- Navigation requests --> Static assets (SPA HTML/JS/CSS)
  |-- fetch("/api/*")     --> Worker (Hono router)
                                |
                                |-- Better Auth (session/OTP management)
                                |-- D1 via Drizzle ORM (users, events, registrations, attendance)
                                |-- Resend API (OTP emails)
                                |-- Web Crypto API (HMAC signing for QR tokens)
```

### Why Single Worker, Not Separate Projects

1. **Simplicity** -- one `wrangler.jsonc`, one deploy, one D1 binding. No service-to-service communication overhead.
2. **Cloudflare Vite plugin** is GA and designed exactly for this: React SPA + Worker API in the same Vite build. The Worker handles `/api/*`; everything else serves the SPA.
3. **Shared types** -- TypeScript types for API responses, user models, etc. are importable across client and server without a monorepo tool.
4. **Cost** -- navigation requests that serve static assets are free (do not invoke the Worker). Only `/api/*` calls are billed.
5. **No CORS** -- same-origin routing means no CORS configuration needed. The SPA and API share the same domain.

### Why NOT separate projects

- The landing page is simple (no SSR, no dynamic data). Adding a Worker API alongside it is trivial with the Vite plugin.
- Splitting into Pages + separate Worker would mean CORS configuration, separate deploys, separate bindings -- unnecessary complexity for this scale.

## Component Boundaries

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| **React SPA** (`src/`) | UI rendering, client-side routing, QR display, camera scanning | Worker API via `fetch("/api/*")` |
| **Hono API** (`worker/`) | Request routing, middleware chain, business logic | Better Auth, Drizzle, Resend, Web Crypto |
| **Better Auth** | Session management, OTP flow, user CRUD | D1 (via native D1 binding), Resend (via callback) |
| **Drizzle ORM** | Type-safe queries, schema-as-code, migrations | D1 Database |
| **D1 Database** | Persistent storage | Accessed only by Drizzle and Better Auth |
| **Resend** | Email delivery for OTP codes | Called by Better Auth callback |
| **Web Crypto API** | HMAC signing (QR tokens) | Used by Worker `lib/qr.ts` |

### Boundary Rules

- **The SPA never accesses D1 directly.** All data flows through `/api/*` endpoints.
- **The Worker never renders HTML.** It returns JSON only (except the SPA fallback handled by static assets).
- **Admin endpoints are separated by middleware**, not by a separate Worker or service.
- **The landing page (`/`) remains a purely static React component** -- no API calls needed.
- **Better Auth handles its own tables** (user, session, verification). App-specific tables are managed by Drizzle.

## Data Flow

### 1. User Registration & Login (OTP via Better Auth)

```
User enters email
  --> authClient.emailOtp.sendVerificationOtp({ email, type: "sign-in" })
  --> POST /api/auth/email-otp/send-verification-otp
  --> Better Auth generates 6-digit OTP, stores in DB (hashed/encrypted per config)
  --> Calls sendVerificationOTP callback
  --> Callback fires Resend email (fire-and-forget, no await)
  --> User enters OTP
  --> authClient.signIn.emailOtp({ email, otp })
  --> POST /api/auth/sign-in/email-otp
  --> Better Auth validates OTP (timing-safe, max 3 attempts, 5min expiry)
  --> Creates/upserts user + creates session
  --> Sets session cookie (HttpOnly, Secure, SameSite)
  --> SPA receives session data
```

### 2. Event Registration & QR Code

```
User views event detail page
  --> GET /api/events/:id
  --> Returns event data

User clicks "Register"
  --> POST /api/events/:id/register
  --> Worker validates: user authenticated? event exists? not already registered? not full?
  --> Worker checks loyalty: SELECT COUNT(*) FROM registrations WHERE user_id=? AND checked_in=true
  --> If count % 5 == 4, mark is_free_entry = true
  --> Worker generates QR token: HMAC-SHA256(registrationId + eventId + userId, QR_SECRET)
  --> Stores registration in D1 via Drizzle
  --> Returns { registration, qrToken }

User views "My Ticket" page
  --> SPA renders QR code from stored qrToken via qrcode.react
  --> QR encodes: "dozis:{token}" (simple prefix + opaque token)
```

### 3. QR Check-In (Admin)

```
Admin opens Scanner page
  --> Camera scans QR code from attendee's phone (html5-qrcode)
  --> SPA decodes QR: extracts token after "dozis:" prefix
  --> POST /api/check-in { token }
  --> Worker looks up registration by token
  --> Worker recalculates HMAC and compares (timing-safe)
  --> Checks: not already checked in? correct event (today's)?
  --> Updates registration: checked_in = true, checked_in_at = now
  --> Returns { success, attendeeName, loyaltyCount }
  --> Admin sees green flash + attendee name
  --> On duplicate: returns "already checked in" with original check-in time
```

### 4. Loyalty Tracking ("5th Event Free")

```
At registration time:
  --> Worker counts user's verified attendances (checked_in = true)
  --> If count % 5 == 4 (this will be the 5th in cycle)
  --> registration.is_free_entry = true
  --> Display "Ez az event INGYENES!" on ticket

At check-in time:
  --> Registration marked as checked_in = true
  --> Next registration will recalculate count
```

## D1 Schema Design (Drizzle)

```typescript
// worker/db/schema.ts
import { sqliteTable, text, integer, uniqueIndex } from 'drizzle-orm/sqlite-core';

// Better Auth manages its own tables automatically:
// - user (id, email, name, createdAt, updatedAt, ...)
// - session (id, userId, expiresAt, token, ...)
// - verification (id, identifier, value, expiresAt, ...)

// App-specific tables:

export const events = sqliteTable('events', {
  id: text('id').primaryKey(),              // nanoid
  title: text('title').notNull(),
  description: text('description'),
  date: text('date').notNull(),             // ISO 8601 datetime
  time: text('time'),
  location: text('location').default('DOPAMIN, Budapest'),
  lineup: text('lineup'),                   // JSON array of DJ names
  genreTags: text('genre_tags'),            // JSON array of genre strings
  capacity: integer('capacity'),            // NULL = unlimited
  imageUrl: text('image_url'),
  status: text('status').default('draft'),  // draft | published | cancelled | past
  createdAt: text('created_at').notNull(),
  updatedAt: text('updated_at').notNull(),
});

export const registrations = sqliteTable('registrations', {
  id: text('id').primaryKey(),              // nanoid
  userId: text('user_id').notNull(),        // references better-auth user.id
  eventId: text('event_id').notNull().references(() => events.id),
  qrToken: text('qr_token').notNull().unique(),
  isFreeEntry: integer('is_free_entry', { mode: 'boolean' }).default(false),
  checkedIn: integer('checked_in', { mode: 'boolean' }).default(false),
  checkedInAt: text('checked_in_at'),
  checkedInBy: text('checked_in_by'),       // admin user.id who scanned
  status: text('status').default('active'), // active | cancelled
  createdAt: text('created_at').notNull(),
}, (table) => ({
  userEventUnique: uniqueIndex('idx_reg_user_event').on(table.userId, table.eventId),
}));
```

### Schema Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Better Auth owns user/session tables** | No need to manually manage users, sessions, OTP codes. Better Auth creates these tables on first run. |
| **No separate attendance table** | `checked_in` boolean + `checked_in_at` timestamp on registrations is sufficient. A user registers then optionally gets checked in. One table, clear state. |
| **UNIQUE(user_id, event_id)** | Prevents duplicate registrations at the database level. The INSERT fails, no application-level race condition possible. |
| **UNIQUE(qr_token)** | Fast lookup for QR scan validation. One index scan. |
| **TEXT dates (ISO 8601)** | D1/SQLite stores dates as text. ISO 8601 sorts correctly. |
| **Loyalty count is derived** | `SELECT COUNT(*) FROM registrations WHERE user_id=? AND checked_in=true`. No denormalized counter to keep in sync. At DOZIS scale (100-300 users), this query is instant. |
| **Admin list in env vars, not DB** | Per PROJECT.md, 3 fixed admins. `ADMIN_EMAILS` as comma-separated env var / secret. |

## Auth Architecture (Better Auth)

### Why Better Auth, Not Custom JWT

The previous architecture draft used custom JWT via Web Crypto. Updated recommendation: **use better-auth**.

| Concern | Custom JWT | Better Auth |
|---------|-----------|-------------|
| OTP flow | Manual: generate, hash, store, validate, expire, rate-limit | Built-in plugin: configurable length, expiry, attempts, storage mode |
| Session management | Manual: JWT creation, cookie setting, refresh logic | Built-in: HttpOnly cookies, session table in D1, revocation |
| Brute force protection | Manual: rate limit per email, attempt counting | Built-in: `allowedAttempts`, automatic invalidation |
| Race condition on OTP | Manual: timing-safe comparison, atomic invalidation | Built-in since v1.5.6: prevents OTP reuse via race condition |
| User enumeration | Manual: constant-time responses | Built-in: `disableSignUp` option prevents enumeration |
| D1 integration | Manual: raw queries or Drizzle adapter | Native D1 support since v1.5 |
| Hono integration | Manual: middleware chain | Documented on hono.dev with examples |
| Code to write | ~300-500 lines of auth logic | ~50 lines of config |

### Better Auth Server Setup

```typescript
// worker/lib/auth.ts
import { betterAuth } from 'better-auth';
import { emailOTP } from 'better-auth/plugins';
import { Resend } from 'resend';

export function createAuth(env: Env) {
  return betterAuth({
    database: { type: 'd1', d1: env.DB },
    plugins: [
      emailOTP({
        otpLength: 6,
        expiresIn: 300,        // 5 minutes
        allowedAttempts: 3,
        sendVerificationOTP: async ({ email, otp, type }) => {
          const resend = new Resend(env.RESEND_API_KEY);
          // Fire and forget -- no await prevents timing attacks
          resend.emails.send({
            from: 'DOZIS <noreply@dozis.hu>',
            to: email,
            subject: `${otp} -- DOZIS belepesi kod`,
            html: `<p>A belepesi kodod: <strong>${otp}</strong></p>`,
          });
        },
      }),
    ],
  });
}
```

### Better Auth Client Setup

```typescript
// src/lib/auth-client.ts
import { createAuthClient } from 'better-auth/client';
```

### Mounting in Hono

```typescript
// worker/routes/auth.ts
import { Hono } from 'hono';
import { createAuth } from '../lib/auth';

const authRoutes = new Hono<{ Bindings: Env }>();

authRoutes.all('/**', async (c) => {
  const auth = createAuth(c.env);
  return auth.handler(c.req.raw);
});

export { authRoutes };
```

**Critical:** Create the auth instance PER REQUEST. D1 bindings only exist within request context on Workers.

```typescript
import { authRoutes } from './routes/auth';
import { eventRoutes } from './routes/events';
import { registrationRoutes } from './routes/registrations';
import { adminRoutes } from './routes/admin';

import { sessionMiddleware } from './middleware/auth';
import { adminMiddleware } from './middleware/admin';

  DB: D1Database;
  RESEND_API_KEY: string;
  QR_SECRET: string;
  ADMIN_EMAILS: string;

const app = new Hono<{ Bindings: Env }>();

// DB middleware for all API routes
app.use('/api/*', dbMiddleware());

// Auth routes (Better Auth handles its own auth)
app.route('/api/auth', authRoutes);

// Authenticated routes
app.use('/api/*', sessionMiddleware());
app.route('/api/events', eventRoutes);
app.route('/api', registrationRoutes);

// Admin routes
app.use('/api/admin/*', adminMiddleware());
app.route('/api/admin', adminRoutes);

export default app;
```

## QR Code Architecture

### QR Token Design

```
Token: HMAC-SHA256(registrationId + ":" + eventId + ":" + userId, QR_SECRET)
QR value: "dozis:{hex-encoded-hmac}"
```

Keep it simple. The HMAC binds the token to a specific registration+event+user tuple. Server validates by recalculating and comparing.

### Token Generation

```typescript
// worker/lib/qr.ts
export async function generateQRToken(
  registrationId: string,
  eventId: string,
  userId: string,
  secret: string
): Promise<string> {
  const message = `${registrationId}:${eventId}:${userId}`;
  const key = await crypto.subtle.importKey(
    'raw',
    new TextEncoder().encode(secret),
    { name: 'HMAC', hash: 'SHA-256' },
    false,
    ['sign']
  );
  const sig = await crypto.subtle.sign('HMAC', key, new TextEncoder().encode(message));
  return Array.from(new Uint8Array(sig)).map(b => b.toString(16).padStart(2, '0')).join('');
}
```

### Token Validation

Recalculate HMAC from stored registration data and compare with timing-safe equality. If the token matches, the QR is authentic. Then check `checked_in` flag for duplicate prevention.

## Patterns to Follow

### Pattern 1: One Drizzle/Auth Instance Per Request

Create both in Hono middleware, store on context. D1 bindings are per-request.

```typescript
// worker/middleware/db.ts
import { drizzle } from 'drizzle-orm/d1';
import * as schema from '../db/schema';

export const dbMiddleware = () => async (c, next) => {
  c.set('db', drizzle(c.env.DB, { schema }));
  await next();
};
```

### Pattern 2: Typed API Client

```typescript
// src/lib/api.ts
async function api<T>(path: string, options?: RequestInit): Promise<T> {
  const res = await fetch(`/api${path}`, {
    ...options,
    credentials: 'same-origin',
    headers: { 'Content-Type': 'application/json', ...options?.headers },
  });
  if (!res.ok) {
    const error = await res.json().catch(() => ({ message: 'Request failed' }));
    throw new ApiError(res.status, error.message);
  }
  return res.json();
}
```

### Pattern 3: D1 Batch for Atomic Operations

D1 does not support `BEGIN/COMMIT`. Use `db.batch()` or Drizzle's batch mode.

### Pattern 4: Admin Check via Email Allowlist

Simple middleware that checks `session.user.email` against `ADMIN_EMAILS` env var.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Global Database/Auth Instances
D1 bindings are per-request. Global instances cause SQLite write lock contention (33-second hangs documented).

### Anti-Pattern 2: Server-Side QR Image Generation
Wastes Worker CPU. QR generation is trivial on client with `qrcode.react`.

### Anti-Pattern 3: Client-Side Auth Validation Only
Hiding admin UI is UX, not security. Server middleware must validate on every admin API call.

### Anti-Pattern 4: Separate Worker for API and Pages for Frontend
One project, one deploy, no CORS. The Cloudflare Vite plugin solves this.

### Anti-Pattern 5: Custom Auth Implementation
Better Auth handles OTP generation, validation, rate limiting, session management, timing-safe comparison, race condition prevention. Don't reimplement.

## Scalability Considerations

| Concern | Current (100 users) | Growth (1K users) | Future (10K users) |
|---------|---------------------|--------------------|--------------------|
| **Database size** | < 1 MB | < 50 MB | < 500 MB (10 GB limit) |
| **D1 reads** | Free tier (5M/day) | Free tier | May need paid ($5/mo) |
| **D1 writes** | Free tier (100K/day) | Free tier | Free tier (events are infrequent) |
| **QR scanning** | 1 admin, sequential | 1-2 admins | Multiple admins, D1 handles fine |
| **Email (Resend)** | Free tier (100/day) | Paid plan needed | Paid plan |
| **Static assets** | Cloudflare CDN, free | CDN, free | CDN, free |
| **Worker cold starts** | None (V8 isolates) | None | None |

**Reality check:** DOZIS is a small Budapest collective. 100-300 users is the realistic ceiling for v1.
